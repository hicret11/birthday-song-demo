_Аудит от 2026-07-07. Ветка `feat/consumer-song-paywall`._

## Итог

Код — в хорошем состоянии и почти готов. `tsc` чистый, платёжный и paywall-путь

## 1. Что проверено и признано надёжным ✅

- **Paywall-гейтинг** (`lib/public-song.ts`): для заблокированной песни с сервера
  стрипаются ВСЕ медиа-URL (raw Suno, highlight-cut, full, video, premium,
  slideshow). Клиент получает только текст + метаданные; аудио идёт лишь через
  gated-роут `/api/share/[id]/preview`, который жёстко режет 15с на сервере.
  Обойти клиентский клэмп нельзя. Есть юнит-тесты.
- **Скачивание** (`/api/share/[id]/download`): при `!unlocked` → `402`. Полный
  медиа-URL не отдаётся до оплаты.
- **Stripe webhook**: проверка подписи, разделение `song_unlock` / venue-подписок,
  best-effort фиксация legal-acceptance, dunning-письма. Premium-рендер вынесен в
  `after()` и никогда не блокирует ответ Stripe.
- **Race-fallback**: если редирект успеха обгоняет webhook, страница шара сама
  верифицирует `session_id` в Stripe и разблокирует — покупатель не застревает на
  paywall.
- **Хранилище**: `lib/r2.ts` чисто заалиасен на **Vercel Blob**
  (`BLOB_READ_WRITE_TOKEN`). Никаких «мёртвых» R2-вызовов — заметка о нестыковке
  R2/Blob устарела.
- **Пайплайн генерации**: circuit-breaker, rate-limit (fixed-window + team-bypass),
  spend-cap с дедупликацией алертов, moderation-гейт, structured-output для
  лириков, аккуратная очистка временных файлов. Ошибки refine/whisper/phonetic
  обёрнуты в try/catch с корректными фоллбэками.

## 2. Недостатки, найденные аудитом

**На уровне кода (минорные, не блокеры):**

- **Moderation fail-open**: при отсутствии/невалидном `OPENAI_API_KEY`
  модерация имён/заметок молча пропускает всё (`lib/moderation.ts`). Это осознанный
  дизайн (не блокировать реальные дни рождения), но в проде ключ обязан быть
  валиден — иначе на публичные страницы шара могут попасть непроверенные тексты.
  Чеклист уже отмечает, что старый ключ 401'ил и тихо отключал модерацию + Whisper.
  _Рекомендация: добавить алерт/health-check на «модерация неактивна в проде»._
- **Spend-cap fail-open**: `recordSpendCents` — fire-and-forget (`void`). Если KV
  недоступен, дневной лимит расходов не срабатывает. Приемлемо, но стоит знать.
- **highlight-cut**: при неудачном ffmpeg-probe длительность допускается = 150с.
  Best-effort, деградация качества в редких случаях.

**Операционные (реальные блокеры — только вы):**

- **Ротация утёкших токенов** (§B0 чеклиста): за сессии в терминал попали Vercel
  ×3, Supabase service-role, Stripe test — их надо перевыпустить. _В git их нет и
  никогда не было (проверено), но ротация всё равно нужна._
- **Vercel Pro**: Hobby — non-commercial, и почасовой cron `recover-previews`
  (`0 * * * *`) им отклоняется. Нужен Pro под командой GMT LLC.
- **Stripe LIVE-продукты**: сейчас существуют только test-продукты. Нужны live
  цены (6 шт.), `sk_live_…`, live-webhook, активация платежей на аккаунте GMT LLC.
- **Email-доставляемость**: SPF/DKIM/DMARC для singmybirthday.com в Resend.
- **Remotion-воркер** (качество, можно после запуска): пока не задеплоен, шары
  получают простое 16:9 ffmpeg-видео.
- **Legal**: сверить сущность/дату в `lib/legal.ts` (GLOBAL MOBILITY
  TECHNOLOGIES LLC) перед запуском.

## 3. Что я исправил в этой сессии

- **Добавлены security-заголовки** (`next.config.ts` → `headers()`): HSTS,
  X-Content-Type-Options, X-Frame-Options: SAMEORIGIN, Referrer-Policy,
  Permissions-Policy (микрофон=self, остальное off). Раньше их не было.
  _Сознательно без строгого CSP — он требует пер-страничного allowlist для
  Stripe/Sentry и легко ломает checkout; это отдельная задача._
- **`.gitignore`**: добавлены `public/_test/` (3.7 МБ тестовых mp3, которые иначе
  публично отдавались бы по `/_test/*`) и `scripts/dev/`. Теперь git-статус чист.
- **README**: заменён дефолтный шаблон create-next-app на реальное описание
  стека, путей и порядка запуска.
- **GO-LIVE-CHECKLIST**: устаревшие `R2_*` переменные заменены на
  `BLOB_READ_WRITE_TOKEN` (актуальное хранилище).

Проверки после правок: `tsc --noEmit` — чисто, `eslint next.config.ts` — чисто.

_Примечание: `npm test` не удалось прогнать в этой песочнице — vitest v4 тянет
нативный бинарник rolldown под linux-arm64, которого здесь нет. Это ограничение
среды, не проблема кода; по чеклисту на вашей машине тесты проходят 13/13._

## 4. Рекомендуемый порядок до запуска

1. Ротировать токены (§B0) → обновить в Vercel env.
2. Vercel → Pro под GMT LLC.
3. Создать Stripe LIVE-продукты + live-webhook, активировать платежи.
4. Проверить полный набор env в Production scope (особенно валидный `OPENAI_API_KEY`).
5. Смёржить PR `feat/consumer-song-paywall` → `main`, задеплоить.
6. Smoke-test на проде: генерация → лог `cut≈55s`, moderation без 401; locked-шар
   отдаёт только preview, download=402; одна реальная покупка → unlock → возврат.
7. SPF/DKIM/DMARC в Resend.
8. (После запуска) задеплоить Remotion-воркер для premium-видео.
